# これは音楽用 DSP の算術ライブラリです
目的の関数は sin, cos, log, exp, pow です。
各関数は constexpr であることが前提です。
底が既知の場合は constexpr として計算されるように最適化すること。
inline 展開はフル活用すること。
ランタイムで std より速く(floatなら16bit精度, double なら24bit精度)、コンパイル時はまずまずな速度である程度精度を高くしようという感じです。
C++20までの機能を使っていいです。
割り算はコンパイル時計算で掛け算にできるようならコンパイラが展開しやすいようにしてください。
SIMDが使えそうな部分では使ってください。各環境で動くようにインクルードガードを適切に行うこと。
アルゴリズムも変更していいです。
方針は次の通りです。

1. まずは解析的努力により計算が必要な変域を狭めてなるべく constexpr に逃がす
2. テイラー展開を使うのではなく、L1キャッシュに乗る程度のLUTを使って補間する。
  - その際、必ずしも LUT の取り方が線形であるのが望ましいとも限らないことに注意すること
  - その際、線形補間で精度が間に合わなかったり LUT が膨れすぎるようなら2次近似その他のアルゴリズムを使うこと

## 解析的努力の例

e^x = 2^{x*logx(e)} を使って指数関数を constexpr * ビットシフト * 半径0.5の近似 にする
log(x) を log2 に帰着し(底の変換公式(変換はなるべく constexpr に逃がす)), 仮数部と指数部を分ける (対数法則), メビウス変換を使って 近似の直径を 1/3 まで減らす

## 原則について
私が書いた原則より精度・速度的に望ましい方法があるのだとしたら、また実験により確認できたのであれば、この原則よりそちらを優先してください。

## 実装レポート (2026-07-08)

`tests/compare_std.cpp` で精度・速度を計測し、目標精度（float: 相対誤差
≤2⁻¹⁶≈1.5e-5、double: ≤2⁻²⁴≈6e-8）に対して過剰精度だった箇所の LUT・多項式
を間引き、アルゴリズムを改良した。以下、次にこのファイルを触る AI・人間の
両方が状況を把握できるようにまとめる。

### やったこと

1. **過剰精度の間引き**（誤差が目標の何倍余裕があるかを確認してから削減）
   - `sin`/`cos`(double) LUT: 512→256 エントリ（8KB→4KB）。誤差 9e-10→1.5e-8
     で、まだ 4 倍の余裕。
   - `sin`(float) LUT: 1024→256 エントリ（4KB→1KB）。誤差はほぼ不変
     （5.4e-6→5.8e-6）——float 側の誤差はテーブルサイズでなく位相計算の float
     丸めが支配的だったため、テーブルを 1/4 にしても精度が落ちない。
   - `exp`(float) の Horner 多項式: 6 次→5 次。
   - `log2`(double) の級数: 8 項→7 項。
   - `log2`(float) の級数: 5 項→4 項。ただし単純に削ると打ち切り誤差が出るので、
     最終係数を `1/9`→`0.1543` にチューニングし、削った高次項の誤差を吸収
     させた（ミニマックス的発想）。結果、項を減らしたのに誤差は悪化しなかった
     （1.1e-6→1.0e-6）。
   - **教訓**: LUT やテイラー項数を削るときは、必ず「その関数の誤差の支配項は
     何か」を先に特定すること。テーブルサイズを削っても他の誤差要因（丸め等）
     が支配的ならほぼタダで削減できる。逆に係数チューニングは項数を削っても
     精度を維持できる場合がある。

2. **アルゴリズム改良: `exp`/`pow`(double) 用の共有 `2^t` コア**
   - `detail::exp2_scaled_core` / `exp2_lut_double` を新設。
     `2^t = 2^n · 2^(j/64) · e^g` を 64 エントリ（512B）の LUT + 3 次多項式の
     端数補正で計算する。
   - **バイアス・トリック**: floor 計算にデータ依存分岐を使うと大幅に遅くなる
     （実測で 2.7ns→3.9ns に悪化して原因を特定した）。`t*64 + 65536` のように
     常に正の値へオフセットしてから int64 に truncate すると、truncate が
     floor と一致するため分岐が消える。同種の「符号付き floor」を扱うコードを
     書くときはこのトリックを検討すること。
   - `exp(double)` は `x·(log2e·64)+bias` を FMA 1 回でこのコアに直結。
   - `pow(double)` のオーバーロードを追加し、`2^(e·log2 b)` の融合経路にした。
     従来の汎用 `pow` (`exp(e·ln b)` 経由) にあった `ln ↔ log2` 変換の乗算
     2 回を排除。
   - **既知の限界**: `pow(double)` の相対誤差は 3.1e-8 で目標 6e-8 に対し
     余裕は 2 倍しかない。`log2(double)` の誤差は `pow` で指数 `e` 倍に増幅
     されるため、**`log2(double)` をこれ以上間引いてはいけない**。

3. **計算機的改良: MSVC のインライン予算切れを発見・修正**
   - 大きな翻訳単位（`compare_std.cpp` 全体）では、`pow`/`log2` のような
     小関数でも MSVC のインライン予算が尽きてインライン化されないことがある。
     これが起きると **ランタイムの LUT パスだけでなく、底が既知の場合の
     コンパイル時定数畳み込みも黙って無効化される**（このライブラリの目的の
     根幹に関わる）。
   - 単体の小さい翻訳単位でのマイクロベンチマーク（`pow(2.0, x)` が 2.7ns）と、
     `compare_std.cpp` 内での計測（同じ呼び出しが 7.2ns）を比較して発見した。
     速度が想定より悪いのに理由が分からないときは、関数を切り出した最小
     TU で再計測して比較するとよい。
   - `dspmath.hpp` 冒頭に `DSPMATH_INLINE`（MSVC: `__forceinline`、GCC/Clang:
     `inline __attribute__((always_inline))`）マクロを定義し、`detail`
     ホットパス関数と公開 API（`sin`/`cos`/`log`/`log2`/`exp`/`pow`）全てに
     付与した。これにより `pow2`(float) は 5.0ns→2.7ns、`pow2`(double) は
     7.2ns→2.8ns に改善。単一の変更としては今回最大の効果だった。

### 現在のベンチマーク結果（このマシン、MSVC `/O2 /arch:AVX2 /fp:fast`、noisy ±0.05）

`ratio = dsp_ns / std_ns`（小さいほど speed）

| 関数 | float | double |
|---|---|---|
| sin/cos | ~0.71 | ~0.60 |
| log | ~0.98 | ~0.74 |
| log2 | ~0.95 | ~0.46 |
| exp | ~1.0（互角） | ~0.83 |
| pow | ~0.86 | ~0.62 |
| pow（底2） | ~0.51 | ~0.26 |

`exp`(float) のみ std とほぼ互角（この環境の std AVX2 実装が非常に速いため。
過去の実験で LUT 化を試みたが速度で勝てなかった。詳細は git 履歴・過去セッシ
ョンのメモリ参照）。それ以外は全関数が `std` 以下の実行時間で、精度は全て
目標の 16bit/24bit 予算内。

### 次にやるなら

- `log2(double)` はこれ以上削らない（上記の理由）。
- SIMD 化（関数内部の手動ベクトル化）は過去に検証済みで、このベンチマーク
  （スループット律速、呼び出し側でパイプライン化される）には効かないと分かって
  いる。むしろ遅くなる。再挑戦しないこと。呼び出し側ループの自動ベクトル化を
  阻害しない（分岐レス・インライン可能を保つ）のが正しい SIMD 戦略。
- 新しい関数やパスを追加するときは、ホットパスに `DSPMATH_INLINE` を付け忘れ
  ないこと。忘れると大きな TU でインライン化が失敗し、速度・定数畳み込みの
  両方が静かに壊れる。
